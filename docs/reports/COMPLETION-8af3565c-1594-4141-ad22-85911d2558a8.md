## Root Cause Analysis

The recurring failure `build_error: Max retries exhausted at spawn time` was caused by a **TypeScript build error** in 3 NPS API routes:

```
./app/api/nps/dismiss/route.ts:35:35
Type error: Property 'user' does not exist on type 'never'.
```

### Why It Happened

All three NPS routes (`/api/nps/dismiss`, `/api/nps/prompt-status`, `/api/nps/submit`) used:

```typescript
const supabase = createClient(supabaseUrl, supabaseAnonKey)
const { data: { session }, error: sessionError } = await supabase.auth.getSession()
if (sessionError || !session?.user) { ... }
```

However, the project uses a **custom PostgREST client** (`lib/db.ts`), not the real Supabase SDK. The `createClient()` factory always returns:

```typescript
auth: {
  getSession: async () => ({ data: { session: null }, error: null }),
}
```

TypeScript strict mode infers `session` as type `null`. Then `session?.user` — accessing `.user` on the literal type `null` — resolves to type `never`, causing the error `Property 'user' does not exist on type 'never'`.

This caused the **Next.js production build to fail**, which cascaded into agents unable to spawn (build validation fails before deployment).

---

## Fix Applied

Replaced `getSession()` with the project-standard **JWT auth-token cookie** pattern, consistent with all other authenticated API routes (e.g. `/api/auth/me`, `/api/auth/trial-status`):

```typescript
import jwt from 'jsonwebtoken'
const JWT_SECRET = process.env.JWT_SECRET || 'your-secret-key-change-in-production'

const authToken = request.cookies.get('auth-token')?.value
if (!authToken) { return 401 }

const payload = jwt.verify(authToken, JWT_SECRET) as { userId?: string; id?: string }
const agentId = payload.userId || payload.id
```
