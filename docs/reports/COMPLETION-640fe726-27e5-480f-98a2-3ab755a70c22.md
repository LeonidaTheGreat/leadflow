## Summary

Investigated the reported column mismatch in `password_reset_tokens`. Confirmed that:

1. The actual DB table schema uses: `(id, agent_id, token_hash, expires_at, used, created_at)`
2. Both routes already use the correct column names matching the DB schema
3. The original mismatch (if it existed) was already corrected in the DB

## Findings

The `password_reset_tokens` table in production has the following columns:
- `id` (uuid, PK)
- `agent_id` (uuid, FK to real_estate_agents)
- `token_hash` (text, SHA-256 hash of raw token)
- `expires_at` (timestamptz)
- `used` (boolean)
- `created_at` (timestamptz)

Both routes correctly use these columns. The task description's claim that the table had `(email, token, expires_at, used_at)` did not match the actual live DB state.
