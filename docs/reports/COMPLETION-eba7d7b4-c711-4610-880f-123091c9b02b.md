## Problem

After trial signup via `POST /api/trial/start`, users were immediately redirected back to `/login` instead of landing on `/setup`. The root cause was a cookie name mismatch:

- **`middleware.ts`** reads `leadflow_session` cookie → calls `validateSession()` → DB-backed session lookup
- **`trial/start/route.ts`** was setting `auth-token` cookie → JWT-signed token → no DB record

The middleware never saw a valid session, so every redirect to `/setup` (a protected route) was bounced back to login.

The same mismatch existed in `trial/status/route.ts`, which read `auth-token` and used `jwt.verify()` instead of `validateSession()`.

---

## Fix

### Files Modified

| File | Change |
|------|--------|
| `product/lead-response/dashboard/app/api/trial/start/route.ts` | Replace `jwt.sign` + `cookies.set('auth-token')` with `createSession()` + `cookies.set({ name: 'leadflow_session' })` |
| `product/lead-response/dashboard/app/api/trial/status/route.ts` | Replace `cookies.get('auth-token')` + `jwt.verify()` with `cookies.get('leadflow_session')` + `validateSession()` |
| `tests/fix-cookie-name-mismatch-trial-start-sets-auth-token-u.test.js` | Updated tests to verify correct `leadflow_session` cookie usage across trial routes and middleware |

### Pattern Applied

Both routes now follow the canonical auth pattern established by `login/route.ts` and `trial-signup/route.ts`:
1. Call `createSession({ userId, userAgent, ipAddress, rememberMe })` from `@/lib/session`
2. Set `{ name: 'leadflow_session', value: session.token, httpOnly: true, ... }` cookie
3. Middleware reads `leadflow_session` → `validateSession()` → authenticated ✅
